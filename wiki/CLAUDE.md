# Wiki Conventions

This `wiki/` folder is an Obsidian vault — a human-readable knowledge graph of project info, decisions, research, design notes, contacts, and daily logs. It is *separate* from:

- Claude built-in project memory at `~/.claude/projects/<mapped>/memory/` — atomic feedback rules + identity, auto-injected by harness
- `~/.claude/memory/` — global cross-project tool gotchas, auto-injected once per session by SessionStart hook
- `docs/` — formal project docs, PRDs, specifications, public-facing artefacts

If you (Claude) are unsure where info belongs, prefer `wiki/` for: project state, decisions and rationale, lessons learned, half-formed thoughts, links between concepts, daily progress notes. Cross-cutting feedback rules → built-in. Cross-project tool gotchas → global memory. Formal/public artefacts → `docs/`. See [[decision-collapse-memory-to-wiki]] for the layer split rationale.

## Folder layout

| Folder | Contents | Filename pattern |
|---|---|---|
| `wiki/` (root) | `index.md` only | `index.md` |
| `wiki/daily/` | One file per day, free-form log | `YYYY-MM-DD.md` |
| `wiki/notes/` | Evergreen atomic notes — one concept per file | `kebab-case-title.md` |
| `wiki/research/` | External references, articles, videos, papers — each summarised with source link | `kebab-case-slug.md` |
| `wiki/people/` | Person profile per file | `firstname-lastname.md` or handle |
| `wiki/projects/` | Sub-project / experiment trackers | `kebab-case-name.md` |
| `wiki/mocs/` | Maps of Content — index notes that link to clusters of related notes | `area.md` (e.g. `game-design.md`) |

## Naming

- All filenames: lowercase, kebab-case, `.md` extension. Example: `chess-narrative-engine.md`.
- Note titles inside the file: H1 in Title Case. Example: `# Chess Narrative Engine`.
- Daily notes: `YYYY-MM-DD.md` (no title needed, date is the title).
- Session test plans: `test-plan-[date]-[topic].md` in `notes/` — date before topic, e.g. `test-plan-2026-07-10-personas.md` (user 2026-07-10).
- Test-plan body: numbered `## N.` sections; testable items as numbered check boxes (`1. [ ]` — Obsidian renders numbered task lists) so the user can tick what's tested AND reference results as `section.item` ("7.4"). Info-only sections (known/parked, next-batch heads-ups) get plain bullets or plain numbers, no boxes (user 2026-07-10).

## Linking

- Use Obsidian wikilinks: `[[note-name]]`. No `.md` extension in the link.
- Link eagerly. If a concept is mentioned twice, make it a note and link both.
- Linking across folders works without prefix as long as note basenames are unique. If a name collides, use `[[folder/note-name]]`.
- Backlinks are auto-generated by Obsidian — don't maintain them manually.

## Tags

Tags encode **orthogonal axes that folders don't capture**: domain, lifecycle, scope, origin. They do *not* duplicate folder structure (no `#decision` — use `path:notes/decision-*` for that).

Tags live in YAML frontmatter at the top of a note:

```yaml
---
tags:
  - domain/supabase
  - domain/security
  - status/adopted
  - scope/m1
  - origin/v1-failure
---
```

Inline `#tag` in body also works — Obsidian indexes both. Always use the namespace (`#domain/x`, never bare `#x`).

### Axes

| Axis | Prefix | Values | Cardinality |
|---|---|---|---|
| Domain (what it's about) | `domain/` | `supabase`, `auth`, `security`, `realtime`, `chess-engine`, `narrative`, `vercel`, `ci-cd`, `stack`, `memory` | 1+ per note |
| Status (lifecycle) | `status/` | `adopted`, `deferred`, `superseded`, `verified`, `open`, `draft` | exactly 1 |
| Scope (milestone) | `scope/` | `foundation`, `m1`, `m2`, `m3-plus` | 0–1 |
| Origin (why it exists) | `origin/` | `v1-failure`, `verification-gate`, `external-research` | 0+ |

### Governance rules

- **Promote, don't sprawl.** Add a new tag only if 3+ existing notes would also use it. One-offs → wikilinks instead.
- **One status per note.** Update when state changes.
- **Domain tags are additive.** A note can be `#domain/supabase` AND `#domain/security`.
- **MOCs, daily logs, `index.md`, `.gitkeep` files don't get tags.** Folder already classifies them.
- **Don't tag templates** (e.g. `_template.md`) except with placeholder comments to guide copies.

### Useful queries

- `tag:#domain/security` — every note touching security across folders
- `tag:#scope/m1 tag:#status/adopted` — M1 decision spine
- `tag:#status/deferred` — work pushed to later milestones
- `tag:#origin/v1-failure` — decisions driven by v1 incidents

## Maps of Content (MOCs)

- An MOC is a hand-curated index note. Lives in `wiki/mocs/`.
- Each area gets one MOC: e.g. `mocs/game-design.md`, `mocs/architecture.md`, `mocs/decisions.md`, `mocs/research.md`.
- When you add a new note in `notes/`, add a link to it from the relevant MOC in the same edit.
- Top-level `wiki/index.md` is an MOC of MOCs — links to every MOC and active project page.

## Daily notes

- Created on demand at `wiki/daily/YYYY-MM-DD.md`.
- Free-form. Capture: what happened today, decisions made, open questions, things to revisit.
- When something in a daily note becomes worth keeping long-term, lift it into `notes/` (or update the relevant MOC) and link to it from the daily note.

## Decisions

- Significant decisions go in `notes/` as atomic notes prefixed with `decision-`. Example: `notes/decision-supabase-vs-firebase.md`.
- Each decision note: context, options considered, what was chosen, why, date.
- Linked from `mocs/decisions.md`.

## When you (Claude) should write to the wiki

- User asks to "write this down", "save this", "remember this for later" — and it's not a memory-system fact (preferences, conventions, identity).
- A decision is made — capture as `notes/decision-*.md`.
- User describes a person, contact, collaborator — `people/*.md`.
- New sub-project or experiment kicks off — `projects/*.md`.
- During work that feels notable — append to today's daily note.

## When NOT to use the wiki

- Cross-cutting feedback rules I need every turn → Claude built-in project memory (`~/.claude/projects/<mapped>/memory/feedback_*.md`)
- Cross-project tool/language gotchas → global memory (`~/.claude/memory/tools/` or `domain/`)
- Formal specs, PRDs, public docs → `docs/`
- Code or code comments → in the code

## Edits and links

When you create a new note, also:
1. Add a wikilink to it from the relevant MOC.
2. If the MOC didn't exist yet, create the MOC and add it to `index.md`.
3. If today's daily note exists, append a one-line reference: `- created [[note-name]]`.
